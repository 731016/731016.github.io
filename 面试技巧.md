## 业务主要流程

### 离散制造MRP负责的部分业务



通过项目计划对产品进行统一管理，通MRP运算



![image-20260404212614778](https://note-1259190304.cos.ap-chengdu.myqcloud.com/noteimage-20260404212614778.png)

**MRP实际运算流程**



第一步：毛需求计算



毛需求是指不考虑任何库存的理论总需求量。



顶层（最终产品）：毛需求直接来源于MPS



下层（组件/原材料）：毛需求 = 父项物料的计划订单下达数量 × BOM中的用量



第二步：净需求计算



`净需求 = 毛需求 - 现有库存 - 在途订单 + 安全库存 + 已分配量`



第三步：计划订单下达



根据净需求和物料的提前期（Lead Time），进行反向排程（Backward Scheduling）：



计划订单收货日期 = 需求产生的日期

计划订单下达日期 = 计划订单收货日期 - 提前期



![image-20260404213757936](https://note-1259190304.cos.ap-chengdu.myqcloud.com/noteimage-20260404213757936.png)



第四步：逐层展开



将计算出的计划订单作为下一层子项的毛需求来源，重复步骤1至3，直至分解到所有原材料和外购件





计算的出的自制件生成生产订单下发到MES执行生产（生产订单->匹配工艺和BOM->生产准备（物料齐套、订单领料、其他需要的工装等）->工序执行（上料、首检【批量生产前单件质检】、自检、报工、完工检验、合格转下序）->产品入库）



计算出的外购件生成采购计划执行采购（采购计划->供应商选择与询价->采购订单->采购到货->采购检验入库）

### 流程制造MES负责的部分业务

#### 生产消耗

根据生产牌号的BOM信息，机台产量，通过WMS系统的上报的车间物料消耗数据，和机台领用台账，计算物料消耗到机台，比如这个机台消耗量多少物料，产量是多少，机台效率...

#### 库存

生产牌号需要的物料的库存管理，主要负责专注于物料从收货->检验->出库投入生产这个流程的控制
收货需要创建在MES创建入库单，由WMS物流管理平台发送数据到MES，进行收货；MES校验货物信息（货物数量）后，根据物料的检验标准创建报检单进行抽检，抽检合格后入库

#### 质量管控

通过维护牌号工艺标准或生产牌号的原料的检验信息，如烟丝的烘丝温度，接装纸的抗张拉力强度

通过生产工单和检验标准创建检验工单进行检验，现场使用检查仪器，数采数据通过开发的net桌面程序，通过udp、tcp、opc、文件、串口通信上传至检验工单

#### SPC负责的业务

监控数采数据，计算CPK,PPK..质量参数，生成监控图形

通过创建的工艺标准下发的数采底层系统，PLC通过OPC协议上传至Wonderware数据库，包含牌号，批次，工单号等信息

通过kafka采集Wonderware数据库的实时数据，并处理计算需要的数据，通过websocket推送至前台展示实时监控图形





#### 烟草制造生产流程

用户负责对成品烟进行排产，排出生产计划，计划导入mes系统，具体到机台，这个机台今天生产多少箱烟



一个新的牌号



牌号基础信息由中烟主数据下发

用户在mes系统维护牌号的工艺信息（制丝、卷包的质量参数、工序信息下发到底层集控系统）、牌号BOM（生产这些牌号需要用到的卷接、包装材料）、配方（生产烟需要用到的原料烟丝等），这些信息会在数采时使用

用户下发生产指令单（牌号等信息）给底层集控系统，交互使用中间表交互，集控解析信息后返回生产工单信息（包括班次，开始结束时间，牌号，批次，工单号等），生产工单的生产状态（下发PLC，开工，完成）由集控系统通过接口返回



一个工单对应多个工序，每个工序的开工信号由SPC维护，比如多个数据点位（物料流量、烘丝温度...），每个工序会有一个主要点位，根据这个点位的数据来判断是否开工或完工，并进行数采，计算，实时显示监控图形；完工后数据信息也会保存，可查看历史监控信息



产量由产量确认单提供，绑定到工单，部分信息（数采产量）由底层系统提供，部分由用户填写，完工时会走审批流程手动确认



生产消耗

每个机台用了多少材料信息由WMS提供，只有总的材料量；没有记录如果这个机台有多的材料借给别的机台用了，通过用户手动录入，mes再班次结束是自动汇总计算



生产用到的材料通过出入库记录来绑定工单号



物料入库，入库单由WMS接口提供，可能一个单据有多个材料，收货后对应明细都会生成一条单据

收货时，生成材料检验单，提交后会生成检验工单，这里没有堵死，只是记录一个流程，这里收货后单据状态就是完全收货了

这里的物料检验由实验室完成，用户根据检验工单创建质检单据，检测完成后，通过tcp、ucp、文件、opc、串口形式传输到mes系统的检验工单明细

## 遇到的技术难题

服务器无法处理请求，nginx代理其中一台服务器处理不了请求，直接访问也打不开页面

1.日志查看出现GC

```
jmap -dump 查看是否内存溢出
发现对象暴涨 → 怀疑内存不足或大数据量查询
```



2.查看进程占用高达1400%

```
top -Hp

printf %\n

jstack 线程 获取堆栈信息
发现某个方法一直在执行，分析可能是数据量大/数据库死锁
```

这个方法是统计报表用到，每个月底会执行，数据每天大概200w，统计实验室的温度，湿度等数据



每个月定时器会执行一次，查看日志发现执行了2个小时！

用户也可以手动生成，应该是点击了手动生成



本地调试发现，执行数据库查询方法直接卡死



把SQL拿到数据库工具执行，发现执行1分钟左右，确实是数据量大一次性加载到内存导致GC



排查SQL问题

（1）索引优化，增加点位编码和时间的联合索引

（2）select * 修改为具体字段，字段指定jdbc类型和java类型转换，减少类型推断

（3）多线程并行通过点位和时间段分批处理
